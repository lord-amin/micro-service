‫‫مستند راهنمای سطوح دسترسی

از نسخه 5.0.0 به بعد برای مدیریت سطح دسترسی ها ، هر سرویسی به صورت جداگانه باید json ای را به auth-server معرفی کند تا این مولفه بتواند سطح دسترسی ها را بر اساس آن کنترل کند. فرمت و تعریف محتوای این json به صورت زیر است:

"permissions": [
  {
    "name": "نام سطح دسترسی",
    "title": "عنوان سطح دسترسی",
    "desc": "توصیفی از سطح دسترسی",
    "apiList": [
      {
        "method": "نوع متد",
        "url": " ادرس متناظر با سطح دسترسی"
      }
    ]
  }

نام سطح دسترسی باید در کل سیستم یکتا باشد.

نوع متد باید از استانداردهای پروتکل http تبعیت کند.

آدرس متناظر با سطح دسترسی باید همیشه با /api/ آغاز شود.

مثال:

{
  "permissions": [
    {
      "name": "user_management",
      "title": "User Management",
      "desc": "User Management",
      "permissions": [
        {
          "name": "auth_server_user_read",
          "title": "Auth Server User Read",
          "desc": "Auth Server User Read",
          "apiList": [
            {
              "method": "get",
              "url": "/api/users"
            },
            {
              "method": "get",
              "url": "/api/users/**"
            }
          ]
        },
        {
          "name": "auth_server_user_create",
          "title": "Auth Server User Create",
          "desc": "Auth Server User Create",
          "apiList": [
            {
              "method": "post",
              "url": "/api/users"
            }
          ]
        }
      ]
    }
  ]
}

با توجه به مثال بالا سه رکورد در دیتابیس وارد می شود یکی برای  user_management و یکی برای auth_server_user_read  که در دیتابیس لینکی به پدر خود که user_management   باشد ، دارد و یکی هم برای auth_server_user_create که لینکی به پدر خود در دیتابیس دارد.
